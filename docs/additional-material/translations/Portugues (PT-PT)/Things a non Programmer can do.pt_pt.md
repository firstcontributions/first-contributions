# Coisas que não programadores podem fazer
## Começar por ouvir

Tudo em open source envolve outras pessoas. Procuras juntar‑te a uma comunidade e isso implica perceber como ela funciona.

Chegar a um projeto e dizer: "Isto é o que deveriam fazer" não costuma ser bem recebido. Alguns projetos podem tolerar essa abordagem, mas em projetos já estabelecidos é provável que essa atitude não seja aceite.

**Ouvir é a melhor forma de conhecer as necessidades de um projeto**

- **Join a mailing list**: Para muitos projetos, a mailing list é o canal principal de comunicação sobre o desenvolvimento. Em projetos grandes há várias listas. Segue a lista orientada a utilizadores e a lista de desenvolvedores principais para começares a ouvir.

- **Follow a blog**: Blogs de desenvolvedores principais dão informação sobre o que está para vir e como o projeto evolui. Sites agregadores (planet) reúnem várias fontes; procura por "planet <nome do projeto>".

- **Join an IRC channel**: Muitos projetos usam IRC para conversar; confere o site do projeto para detalhes.

Trabalhar com tickets
O código é o núcleo de um projeto open source, mas escrever código não é a única forma de contribuir. A manutenção e os sistemas à volta do código são frequentemente negligenciados. Trabalhar nesses pontos é uma forma fácil de começar.

- **Diagnose a bug**: Bugs são por vezes mal reportados. Diagnosticar e triagem de um bug ajuda a poupar trabalho aos desenvolvedores. Se um utilizador reporta "O software não funciona quando faço X", tenta reproduzir, encontra passos exactos, e adiciona essa informação ao ticket.

- **Close fixed bugs**: Muitas vezes os bugs são corrigidos no código mas os tickets não são actualizados. Fecha tickets que já foram resolvidos.
